L - Liskov Substitution Principle (LSP)

Original Definition
If S is a subtype of T, then objects of type T in a program may be replaced with objects of type S without altering any of the desirable properties of that program.

Simple Definition
The LSP says that if you have a class, you should be able to use any of its subclasses interchangeably without breaking the program.
Example
In the initial example, we have an AudioProcessor class responsible for various audio processing operations such as compression, increasing audio size, separate music/vocals, and enhancing quality with AI.  There's also a LimitedAudioProcessor class that extends AudioProcessor, but it overrides the separateMusicAndVocal and enhanceQualityWithAI methods to throw errors indicating that these features are only available in the premium version.

This violates the Liskov Substitution Principle because substituting an instance of LimitedAudioProcessor for an instance of AudioProcessor could lead to unexpected errors if code relies on those overridden methods.

To adhere to the LSP, we refactor the classes. We create a PremiumAudioProcessor class that extends AudioProcessor and implements the separateMusicAndVocal and enhanceQualityWithAI methods.  This way, both classes share a common interface and substituting an instance of PremiumAudioProcessor for an instance of AudioProcessor won't break the program's correctness.

In the refactored version, AudioProcessor is now focused on basic audio processing operations like compression and audio size, while PremiumAudioProcessor extends it to include premium features like separating music/vocals and enhancing quality with AI.  This separation allows for better code organization and adherence to the Liskov Substitution Principle.
